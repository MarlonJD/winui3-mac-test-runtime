## Goal

Move from PR #3's source-level harness readiness to the first honest
native-quality renderer promotion tranche.

The first target is `component-basic-input-light`, with promotion considered only
for rows whose native Windows crop, macOS runtime crop, pixel diff, provenance,
manual inspection notes, and required interaction/accessibility evidence support
the grade.

The production claim remains bounded:

- no arbitrary WinUI compatibility claim;
- no production visual fidelity claim for unpromoted rows;
- no publish/release permission claim;
- `nativeQualityGrade: not-evaluated` remains valid whenever evidence is not
  strong enough.

## Architecture

The work has three cooperating layers:

1. Renderer fidelity in `skia-v2`, centered on shared Fluent primitives, text
   baseline handling, icon font resolution, and component painters.
2. Evidence generation through strict visual runs, component crops, diffs,
   provenance, dashboards, review index pages, and inspection metadata.
3. Manual review gates that decide whether evidence supports `good` or
   `production-ready` for a specific row.

No layer can promote a row by itself. Automated strict visual evidence can make a
row reviewable; manual inspection can accept bounded visual gaps; dashboard
generation can record the result. Promotion requires all three.

## Assumptions And Open Questions

- Native Windows screenshots remain the visual source of truth.
- A row can remain `usable` for harness purposes while still being blocked for
  native-quality promotion.
- Component crop size mismatches, spacing mismatches, state chrome gaps, and icon
  glyph mismatches are real renderer issues until inspected and accepted.
- Text font mismatch is not automatically a release blocker for the first
  tranche, but it must be measured before native-quality promotion.
- Icon font mismatch is higher risk than body text mismatch for `FontIcon`,
  `SymbolIcon`, command icons, and `RatingControl`.
- Windows font installation on macOS must not become an undocumented hidden
  dependency. If optional local fonts are used, the exact detected font family
  must be recorded in generated evidence.
- Open question: whether the first tranche should target `good` only, reserving
  `production-ready` until state/theme/accessibility evidence is broader.

## Font And Icon Strategy

WinUI icons normally come through icon fonts for `FontIcon` and `SymbolIcon`.
Current Microsoft guidance says those controls use the `SymbolThemeFontFamily`
theme resource; Windows 11 uses `Segoe Fluent Icons`, with older Windows falling
back to `Segoe MDL2 Assets`.

The original `skia-v2` renderer requested `Segoe UI` for both normal text and icon
glyphs. That is acceptable for source-level harness readiness, but it is not a
credible final icon strategy.

Plan:

1. Add a renderer font resolver that separates text and symbol fonts.
2. Text font order:
   - `Segoe UI Variable`, when available;
   - `Segoe UI`, when available;
   - documented platform fallback when unavailable.
3. Symbol font order:
   - `Segoe Fluent Icons`, when available;
   - `Segoe MDL2 Assets`, when available;
   - deterministic vector fallback for only the supported public glyphs, with
     evidence marked as fallback.
4. Record detected text font, detected symbol font, and fallback mode in
   component evidence.
5. Run an A/B strict visual experiment for `component-basic-input-light`:
   - current fallback state;
   - environment with licensed Windows fonts installed locally, if available;
   - vector fallback for supported public icons, if implemented.
6. Do not commit proprietary fonts.
7. Do not require Windows font installation for source-level CI.
8. Require either a real symbol font or reviewed deterministic fallback before
   promoting icon-dependent rows.

Decision rule:

- If Segoe text fonts materially reduce text/layout diffs, add font availability
  to promotion notes and keep fallback rows below final native quality.
- If Segoe text fonts do not materially change first-tranche outcomes, continue
  prioritizing renderer geometry and state chrome.
- If icon fonts materially change icon rows, make symbol font resolution a
  promotion gate for those rows.

## Manual Inspection Gate

Manual inspection is required, not optional.

For every row considered for promotion:

1. Open `docs/visual-parity/public-visual-review-index.html`.
2. Open the scenario-specific `visual-review.html`.
3. Inspect the native Windows crop, macOS runtime crop, and pixel diff crop
   side by side.
4. Compare identity, bounds, size, spacing, corner radius, stroke, fill,
   typography, icon glyph, text baseline, disabled/checked/selected state,
   focus affordance, and interaction evidence.
5. Record reviewer, date, native reference run ID, comparison artifact paths,
   accepted gaps, tolerance reason, and a short note in
   `component-inspection.json`.
6. Apply inspection metadata through runner tooling.
7. Regenerate the dashboard and review index.

A row must not be promoted when:

- the native crop and macOS crop are not comparable;
- the icon font or glyph is visibly wrong;
- text baseline or control size changes the native component identity;
- required state evidence is missing;
- accepted gaps would amount to a production visual fidelity claim the evidence
  does not support.

## Risks And Mitigations

- Risk: font installation hides real layout issues.
  Mitigation: run A/B evidence and separate text rows from icon-dependent rows.
- Risk: proprietary fonts are accidentally committed.
  Mitigation: do not add font binaries; record only detected font family names
  and fallback mode.
- Risk: icon glyphs render from the wrong font and look superficially present.
  Mitigation: make symbol font detection explicit and manually inspect glyph
  identity.
- Risk: renderer primitive changes regress other scenarios.
  Mitigation: keep the first tranche narrow and run dashboard/catalog checks
  after each phase.
- Risk: manual inspection over-promotes approximate rows.
  Mitigation: require notes that tie the grade to visible native/macOS/diff
  evidence and leave uncertain rows `not-evaluated`.

- Keep font resolver changes separate from painter changes where possible.
- If a font experiment does not improve evidence, retain only diagnostics that
  help future reviews.
- If a renderer primitive regresses a non-target scenario, either fix the helper
  or revert that helper change before promotion.
- If inspection metadata over-promotes a row, revert the affected inspection and
