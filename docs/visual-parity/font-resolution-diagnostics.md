# Font Resolution Diagnostics (skia-v2)

Date: 2026-06-04

Scope: Phase 0/Phase 1 of
`docs/plans/2026-06-04-renderer-fidelity-native-quality-promotion-plan.md`.

This note records what the skia-v2 renderer's font resolver detected during the
first native-quality tranche spike. Every claim here is bounded to evidence
generated by this repository on the host described below. No
`nativeQualityGrade` was promoted, no proprietary font was bundled, and
`releaseAllowed`/`publishAllowed` remain `false`.

## What changed

The skia-v2 renderer previously requested a single `Segoe UI` typeface for both
body text and icon glyphs. It now resolves the two roles separately and records
the outcome:

- Text font order: `Segoe UI Variable`, then `Segoe UI`, then the platform
  default text font.
- Symbol font order: `Segoe Fluent Icons`, then `Segoe MDL2 Assets`, then the
  resolved text typeface as an explicit fallback.

Diagnostics are emitted as `fontDiagnostics` in each run's `snapshot.json`,
`visual/visual-run.json`, and `visual/component-evidence.json`. When no symbol
font is installed, the symbol role reuses the resolved text typeface, so rendered
pixels are unchanged from the previous single-typeface behavior; the fallback is
reported honestly rather than hidden.

As of the 2026-06-05 follow-up, each font role also reports provenance through
`resolvedSource` and, when an opt-in repo-external font directory supplies the
typeface, `resolvedPath`. This records where the renderer loaded the font from
without copying or embedding any font file in the repository.

## Which rows actually depend on the symbol font

Inspecting `src/WinUI3.MacRenderer.Skia/SkiaV2SnapshotRenderer.cs`, the symbol
typeface is consumed only by:

- `FontIcon` glyphs,
- `SymbolIcon` glyphs,
- `AppBarButton` command icons (`DrawAppBarIcon`, which has vector fallbacks for
  Save/Refresh and otherwise draws a private-use glyph).

None of these appear in the `component-basic-input-light` first tranche. All 13
first-tranche rows draw their affordances with deterministic vector primitives in
`FluentDrawingPrimitives`, not the icon font:

- check mark (`DrawCheckBox`), radio dot (`DrawRadioButton`),
- chevrons (`DrawChevronDown` for ComboBox / DropDownButton / SplitButton /
  ToggleSplitButton),
- slider (`DrawSlider`), toggle switch (`DrawToggleSwitch`),
- rating stars (`DrawRatingStars`).

So for this tranche the symbol-font absence changes no pixels. `RatingControl`,
which the plan listed as icon-dependent, renders vector stars here and is **not**
blocked by the missing symbol font in skia-v2.

## Decision (plan §"Font And Icon Strategy" decision rule)

- **Symbol/icon font is not a first-order blocker for the
  `component-basic-input-light` first tranche**, because no first-tranche row uses
  the icon-font glyph path. It *is* a first-order blocker for genuinely
  icon-font-dependent rows (`FontIcon`, `SymbolIcon`, `AppBarButton` command
  glyphs) in other scenarios: those require a real symbol font or a reviewed
  deterministic vector fallback before promotion.
- **Text font is not a first-order blocker either, but is a measured secondary
  gap.** Body text resolves to `Helvetica`, not the Windows Segoe text family,
  so glyph shapes and advance widths differ from native. This must be recorded before any
  native-quality promotion, but it is not the dominant first-tranche gap.
- **The dominant first-tranche gaps are geometry, not fonts.** Committed evidence
  (`docs/visual-parity/examples/component-basic-input-light/component-evidence.json`)
  shows large native-vs-runtime crop deltas — for example several `Diagnostic*`
  targets are ~180-190 px wider in the runtime crop, `CheckBox` is 24 px narrower,
  and `ToggleSwitch` is 19 px taller. These are layout/bounds issues that font
  resolution does not address.

## Manual Triptych Notes

Reviewed the checked-in native/macOS/diff triptychs for the requested first
tranche rows on 2026-06-04:

- Button text baselines are legible, but the runtime crop is wider than native
  (`158x32` vs `142x32`), so the diff is dominated by sizing.
- CheckBox and RadioButton glyph identity is recognizable, but native stroke,
  fill, and checked-state chrome differ; the RadioButton selected ring/dot is
  visibly weaker in the runtime crop.
- DropDownButton, Slider, ToggleSwitch, and RatingControl runtime crops include
  row labels or wider row regions that native crops do not include, producing
  full-crop diffs before glyph fidelity can be judged.
- RatingControl stars are deterministic vector primitives in this renderer, not
  symbol-font glyphs. Missing `Segoe Fluent Icons` therefore does not block this
  row; crop bounds and star metrics remain visual gaps.
